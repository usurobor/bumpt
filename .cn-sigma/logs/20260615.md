### Learnings worth canonicalizing at home (CI/CD + Vercel realization)

1. A Vercel **token deploys but can't provision Postgres** — DB connect is a
   one-time dashboard step (Vercel Postgres = Neon under the hood).
2. **`vercel env pull` writes a binary-flagged dotenv** — text parsers (sed/grep/
   python-lines) silently fail to read `DATABASE_URL`. Don't extract it in CI;
   instead apply schema via an app endpoint that uses the runtime DB.
3. **Deployment-hash URLs are auth-gated** (Vercel deployment protection); hit the
   **public production alias** for automated calls.
4. **Split SQL on `;` only after stripping `--` comments** (schema had `;` inside
   comments → truncated statements).
5. The dev sandbox has a **locked egress allowlist** (no api.vercel.com / *.vercel.app)
   and **read-only Actions** (can't dispatch) — drive runs by pushing; do network
   work from the runner, which has open egress.

Recommend home fold the **CLI-deploy-from-gate** model and these gotchas into the
cnos CI/CD convention (bumpt = reference body).
